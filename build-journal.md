# Build Journal — Guitar Chord App

## 2026-07-21 — Standing up the process before writing a line of product code

This was the "set up the company" session — nothing about the app's actual code, everything about the scaffolding a real software team would have in place before building starts.

**Why requirements came before architecture.** The project's own ground rules say to capture the full product vision — including features we won't build for a long time — before locking in any technical decisions. So the very first thing that happened was standing up a "PM" AI teammate whose only job was to interview the founder and write down the answer, not to start designing anything. That interview surfaced something that wouldn't have come up otherwise: a fairly specific idea for a mood/scale-based chord progression generator, which is a meaningfully different (and more ambitious) feature than a generic "song mode," and now that distinction is preserved in the requirements doc instead of getting flattened into something vaguer.

**Why GitHub Issues instead of Azure DevOps Boards.** Both were real options, since Azure is already the hosting plan. The deciding factor was that the code was going to live on GitHub anyway (for portfolio visibility), so keeping tickets in the same place means a recruiter — or future-us — can see decisions and work items sitting right next to the code that resulted from them, instead of splitting the story across two disconnected tools.

**A small but real professional lesson: commit identity.** When we connected the project to git, the machine's default identity was a work email. Public commits are visible to anyone, forever (well, until deliberately rewritten), so a personal portfolio project committing under a work email is the kind of small detail that actually matters. We overrode it locally — just for this project, not globally — rather than either ignoring it or changing every repo on the machine.

**Ticket design as its own skill.** Rather than writing tickets as vague goals ("add search"), each one got a title, a short rationale tied back to the requirements doc, and an acceptance-criteria checklist — the same shape GitHub renders as literal clickable checkboxes with a visible progress count. That format was chosen deliberately: it's lightweight enough for a solo builder, but it's the same underlying discipline real engineering teams use so that a ticket is understandable by someone who wasn't in the room when it was written.

**Treating "building the team" as trackable work, too.** Partway through, it became clear that everything so far — standing up the PM agent, setting up GitHub — had happened as plain conversation, with no ticket trail, even though the project's whole premise is operating like a real company. We fixed that by adding a separate `ai-team` label and backfilling closed tickets for what had already been done, rather than pretending the tracking had been there all along. This keeps two visible tracks: tickets about the *product*, and tickets about *how the company/team itself gets built* — which is exactly the kind of thing a non-technical founder learning to run a software company would want visibility into.

**The wiki as the "long-form" layer.** GitHub repos have three natural places to put information, and each got a distinct job this session: `requirements.md` is the detailed source-of-truth doc, GitHub Issues are individual work items, and the wiki is portfolio-facing narrative — "why this project exists" and "how the team operates" — written for someone who won't read the raw requirements doc. Practically, a GitHub wiki turned out to be its own separate mini git repository that only gets created after the very first page is saved through the browser — command-line tools can't create it from nothing, so that one step had to happen manually before anything could be automated.

**Growing the PM agent's scope deliberately, not by default.** When asked what else a real PM does, the honest list included things like user research, competitive analysis, and go-to-market planning — all real PM work, but all of it requires actual users, competitors, or a live launch to mean anything. Adding those now would have been performative rather than useful, so they were explicitly left out, while prioritization and feature-spec-writing — which *do* have real work to do right now, given 17 open tickets and no code yet — were added. The agent also deliberately still can't touch GitHub directly; it drafts, and the founder/assistant execute after review. That's a considered decision, not a default we forgot to revisit — kept in place for now on the founder's own call, with the door left open to change it later.

**The founder deck.** To make all of the above easy to hand to someone who won't read the wiki, we built a scrollable "briefing" web page — a set of full-screen sections styled like a stage setlist (dark, brass-toned, guitar-adjacent) covering the vision, the product scope, the operating model, the AI team, and current ticket status pulled from the real numbers rather than placeholders. It's hosted as a private link by default, with a one-click share toggle to make it public when the founder is ready to show it to someone.
